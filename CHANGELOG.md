## [Unreleased]

### Added

- **Feature D3 — `syncOrderToPennylaneWorkflow` + `syncOrderToPennylane` orchestrator.** End-to-end order → Pennylane sync: fetch order with relations via `useQueryGraphStep`, pick the captured payment (new `pickCapturedPayment` helper uses Medusa's canonical `captured_at` / `canceled_at` signals), resolve the PSP mapper, run C1 + D1 + D2, persist the outcome in `InvoiceSync`. On failure: `status: "failed"` + truncated `last_error`, original error rethrown unchanged (typed HTTP errors preserved). Three-valued `action` result: `"created"`, `"idempotent"` (Pennylane-side match), or `"already-synced"` (local DB short-circuit). Belt-and-suspenders idempotency via the DB unique on `medusa_order_id` + D2's remote pre-check on `external_reference`. No Pennylane-side compensation (invoices immutable; customers normally permanent). 15 new tests across `pick-payment.unit.spec.ts` (5) + `sync-order.unit.spec.ts` (10). Running D3 against a real DB requires roadmap item B4 (first generated migration) — unit tests validate orchestration logic against mocked service CRUD. See [`docs/sync-order-workflow.md`](docs/sync-order-workflow.md).
- **Feature D2 — `createPennylaneInvoice`.** Async helper that POSTs the D1 payload to Pennylane's `POST /customer_invoices` (Finalized branch) with an idempotent pre-check on `external_reference`. Returns `{invoiceId, externalReference, action: "created" | "idempotent"}` — no PDF URL or invoice_number surfaced; callers that need them can GET `/customer_invoices/{id}` separately. Defensive shape validation on both lookup and create responses; typed HTTP errors from `PennylaneClient` propagate unchanged to the future D3 workflow. 10 new tests. See [`docs/invoice-create.md`](docs/invoice-create.md).

### Fixed

- **C1 — `GET /customers` response key is `items`, not `customers`.** `upsertPennylaneCustomer` previously read `response.customers`, always found `undefined`, and would have thrown against the real API. The fabricated `{customers, metadata}` shape in `openapi-customers-list.json` masked the bug in tests. Fixture rewritten to match the real spec (`{items, has_more, next_cursor}`) and lookup code updated accordingly. Verified against the upstream OpenAPI spec on 2026-04-13.
- **D1 — drop `payment_conditions` from the invoice body.** The field is not in the allowed properties for `POST /customer_invoices` on either `anyOf` branch (Draft or Finalized); combined with `additionalProperties: false` on both branches, Pennylane would have returned 422 on every invoice. `payment_conditions` is a customer-level default, not a per-invoice override; removed from the `PennylaneInvoiceCreatePayload` interface and from `buildInvoicePayload`. Payment tracking relies entirely on Pennylane's auto-reconciliation of the `transaction_reference` block.

### Added

- **Features C1 + C2 (bundled) — Pennylane customer upsert + billing address mapping.** `toPennylaneBillingAddress` (C2) is a pure mapper from Medusa's `OrderAddressDTO` to Pennylane's `{address, postal_code, city, country_alpha2}` shape with required-field enforcement and country-code uppercasing. `upsertPennylaneCustomer` (C1) is the async idempotent upsert: derives `external_reference` (`med_cust_<id>` for registered, `med_order_<id>` for guests per Gemini's SOTA 2026 French-invoicing analysis), looks up via `GET /customers?filter=…`, and creates individual or company based on `billing_address.company` presence. B2B payload includes optional SIREN (`reg_no`) and VAT number from configurable `order.metadata` keys. Customer data always taken from `order.billing_address` to freeze invoice identity at sale time per CGI Art. 289. HTTP errors from the client propagate unwrapped to D3's future retry policy; C1 only throws on input validation + defensive response-shape checks. 32 new tests. See [`docs/customer-upsert.md`](docs/customer-upsert.md).
- **Feature D1 — `buildInvoicePayload`.** Pure function that transforms a Medusa v2 `OrderDTO` + the caller-selected `PaymentDTO` + the caller-resolved `PspMapper` into the exact JSON body Pennylane's `POST /customer_invoices` (Finalized `anyOf` branch) accepts. HT extraction uses the robust `(item.total − item.tax_total) / quantity` formula — works for every combination of `is_tax_inclusive`, discounts, and stacked tax lines. Emits a shipping line per non-zero shipping method. Internally composes D6 reconciliation against the self-consistent line sum; >1-cent drift throws with order + line context. PSP mapper resolves `transaction_reference`; when no ref is produced, applies the `onUnknownPsp` option (`"warn"` → non-fatal warning in the returned array, `"accept"` → silent, `"error"` → throw). Returns `{ payload, warnings }`. Ships alongside two helpers in the same folder: `toMinorUnits` (major-unit number → integer cents) and `unwrapBigNumber` (polymorphic `BigNumberValue` → plain `number` with NaN/Infinity rejection). 39 new tests (29 invoice-payload + 5 big-number + 5 toMinorUnits). See [`docs/invoice-payload.md`](docs/invoice-payload.md).
- **Features D5 + D6 (bundled) — Invoice amount helpers.** Two pure, dependency-free helpers in a new `src/modules/pennylane/invoicing/` subfolder. `centsToPennylaneDecimal(amount, currency = "EUR")` converts minor-unit integers to Pennylane's decimal string (2 decimals for EUR/USD/GBP, 0 for JPY/KRW/…, 3 for KWD/BHD/…, 6 for fractional-cent inputs from D6). `reconcileInvoiceLineTotals(lines, expectedTotalCents)` adjusts the largest line (by `quantity × unitPriceCents`) by `drift / quantity` when drift ∈ {-1, +1}; throws above 1-cent drift. Generic signature preserves caller line-shape types (label, vat_rate, etc.). Immutable — never mutates the input array or elements. 30 new unit tests (17 amounts + 13 reconcile). Pennylane-rounding caveat for fractional-cent lines with quantity > 1 documented; D2's live smoke test will validate. See [`docs/invoice-amount-helpers.md`](docs/invoice-amount-helpers.md).
- **Features P1 + P2 (bundled) — PSP mapper registry with Stripe as the first catalogue entry.** Lazy registry resolves `payment.provider_id` → `PspMapper` at runtime, enabling Pennylane auto-reconciliation via the `transaction_reference` block on finalized invoices. Built-in `stripeMapper` matches `pp_stripe_*` (base + all variants) and extracts the PaymentIntent id from `payment.data.id`. Four new plugin options: `onUnknownPsp` (`"warn"|"accept"|"error"`, default `"warn"`), `providerAliases`, `disableMappers`, `customMappers`. Boot-time validation rejects malformed mappers, unknown alias targets, id collisions, and typo'd disable entries with clear error messages. Registry stores `onUnknownPsp` but does not enforce policy — feature D2 applies it. 36 new tests across three specs (stripe-mapper: 14, registry: 18, service: +4). See [`docs/psp-registry.md`](docs/psp-registry.md). Boot-time auto-detection via `paymentModuleService.listPaymentProviders()` is intentionally deferred (candidate future feature P3).
- **Features A4 + A5 (bundled) — Pennylane spec verification and verified VAT enum.** Mechanically verified the upstream OpenAPI spec for every endpoint this plugin uses. Committed 10 minimal excerpts (`src/modules/pennylane/client/__fixtures__/openapi-*.json`) plus a source-of-truth VAT enum fixture. New [`docs/spec-verification.md`](docs/spec-verification.md) ADR captures six decisions: VAT enum (A4), customers filter syntax, credit-note flow, cursor pagination, `transaction_reference` placement, and reservation of the PSP mapper-registry roadmap slot (P1 + P2). Drift-guard test in `vat-rate.unit.spec.ts` compares `PENNYLANE_VAT_RATES` against the fixture in both directions.
- **Feature A1 — `PennylaneClient` HTTP client.** Fetch-based transport for every future Pennylane call. Bearer-auth header, JSON serialization, `AbortController` timeouts (default 10s, per-call override), typed error mapping into six `MedusaError` subclasses (`PennylaneAuthError`, `PennylaneForbiddenError`, `PennylaneNotFoundError`, `PennylaneValidationError`, `PennylaneServerError`, `PennylaneNetworkError`), structured logging with `requestId` correlation, and token redaction via ES private fields. Exposed on `PennylaneModuleService` via `getClient()` and `healthCheck()` (wraps `GET /me`). Plugin options `apiToken` (required), `baseUrl`, `requestTimeoutMs`. 42 unit tests. See [`docs/http-client.md`](docs/http-client.md).

### Changed

- **`PENNYLANE_VAT_RATES` replaces the placeholder enum.** The 5.5% reduced French rate is `FR_55`, not `FR_055` (the latter would have been rejected by Pennylane). Ships 20 French codes + 8 special statuses. Non-French country codes remain available as plain strings for consumers that need them. Downstream features (D1/D2) now build invoice lines against this verified enum.

### Notes

- Pennylane spec discrepancies documented at foundation time were all resolved in feature A4 (see the Added section above). Kept here as a historical note: the foundation shipped with a placeholder enum and open questions about VAT codes, customers filter syntax, credit notes, and pagination.
- `medusaIntegrationTestRunner({ inApp: true })` requires a host `medusa-config.ts`, which a plugin package does not have. The first integration smoke test will be written against a minimal host fixture as part of the first feature PR rather than in the foundation.
